# DB migration audit — Alembic variant

Audit database migrations written for Alembic (commonly with
SQLAlchemy in FastAPI / Flask / Django-via-Alembic / pure-Python
projects).

**This prompt extends [`core/db-migration-audit.core.prompt.md`](./core/db-migration-audit.core.prompt.md).**
Read the core first for the workflow shape (Step 0 convention
sourcing, Step 1 inventory, Step 2 unsafe-operation catalogue, Step
3 severity model, Step 4 report format, and the Constraints). This
file supplies the Alembic-specific file paths, parsing strategy,
and ORM gotchas.

If pasting into a chat without filesystem access, paste the core
first, then this variant.

---

## Assumed stack

- **Migrations tool**: Alembic.
- **Migration files**: Python files in `alembic/versions/` or
  `migrations/versions/` (or whatever `script_location` in
  `alembic.ini` specifies). Each defines `upgrade()` and
  `downgrade()` functions and uses the `op.*` API.
- **Schema source of truth**: SQLAlchemy models, typically under
  `app/models/` or similar. Alembic compares model state against
  database state during `--autogenerate`.
- **Database**: from `sqlalchemy.url` in `alembic.ini` or from the
  env-loaded URL in `env.py`.

grep -E 'transaction_per_migration' alembic/env.py alembic.ini 2>/dev/null

## §2 — Operation catalogue (Alembic specifics)

The core's catalogue applies. Alembic's `op.*` API maps directly to
SQL operations; review accordingly.

### Common `op.*` calls and their SQL

| `op.*` call | SQL emitted | Catalogue category |
|---|---|---|
| `op.add_column(table, Column(..., nullable=False))` | `ALTER TABLE ... ADD COLUMN ... NOT NULL` — fails on populated table without server-side default. |
| `op.alter_column(table, col, nullable=False)` | `ALTER TABLE ... ALTER COLUMN ... SET NOT NULL` — table scan, ACCESS EXCLUSIVE on Postgres. |
| `op.alter_column(table, col, type_=NewType)` | `ALTER TABLE ... ALTER COLUMN ... TYPE ...` — may rewrite the table. |
| `op.drop_column(table, col)` | `ALTER TABLE ... DROP COLUMN ...` — rolling-deploy breaker. |
| `op.create_index('ix_x', 'tbl', ['col'])` | `CREATE INDEX` — **never** `CONCURRENTLY` from the builder. |
| `op.create_foreign_key(...)` | `ALTER TABLE ... ADD CONSTRAINT FOREIGN KEY` — fails on bad data; check for matching index. |
| `op.execute("RAW SQL")` | Escape hatch; review by the core catalogue. |
| `op.bulk_insert(...)` / `op.execute("UPDATE ...")` | Backfill inside migration. |

### `op.create_index` and `CONCURRENTLY` (Postgres)

The Alembic workaround:

```python
def upgrade():
    op.create_index(
        "ix_users_email", "users", ["email"],
        postgresql_concurrently=True,
    )
```

Plus the migration must be marked non-transactional:

```python
# top of the file
from alembic import context

# Either set transaction_per_migration = False in env.py, or:
def upgrade():
    if context.is_transactional_ddl():
        raise RuntimeError("This migration must run without a transaction.")
    op.create_index(..., postgresql_concurrently=True)
```

Or — more commonly — split the index creation into its own
migration with `op.execute("CREATE INDEX CONCURRENTLY ...")` after
setting `transaction_per_migration = False` in `env.py`.

If you see a non-`postgresql_concurrently` `op.create_index` on
what looks like a hot table, flag it with the explicit workaround
above.

### `op.bulk_insert` / `op.execute("UPDATE ...")` inside `upgrade()`

Alembic makes backfills cheap to write; this is a footgun. A
`bulk_insert` of seed data or an `UPDATE` covering many rows runs
inside the migration's transaction (assuming default
`transaction_per_migration`) — holds locks for the duration.

Flag as ⚠️ HIGH and recommend splitting: schema migration first,
data backfill as a separate script / job that runs outside the
migration system.

### `--autogenerate` blind spots

Alembic's `--autogenerate` does not detect:

- Server defaults (`server_default`) — manually authored.
- Trigger functions, materialized views, partial indexes, custom
  types — must be added with `op.execute("RAW SQL")`.
- Check constraints, in some Alembic / SQLAlchemy versions.
- Column reorderings.
- Comments on columns / tables.

If the migration was generated by `--autogenerate` and the schema
delta includes any of the above, the migration is incomplete. Check
the model delta against the migration.

### Branched migrations / `merge` revisions

If the migration's `down_revision` is a tuple, it's a merge
revision. Verify that both branches it merges are reachable on
`main` — a merge revision that depends on an unmerged branch can't
upgrade cleanly.

## §3 — `downgrade()` review

The core's "Rollback / downgrade review" applies. Alembic-specific:

- ⚠️ **`downgrade()` raises `NotImplementedError`** — explicit
  forward-only; flag for the reviewer to confirm it matches
  policy.
- ⚠️ **`downgrade()` reverses fewer operations than `upgrade()`** —
  partial rollback.
- ⚠️ **`downgrade()` references a constraint by a different name
  than `upgrade()` created** — runtime failure during rollback.
- 💡 **`downgrade()` is a verbatim reverse but the upgrade
  included a backfill** — schema reverses, data does not.

## Constraints (Alembic-specific addenda)

- The `op.alter_column` function combines multiple operations
  (rename, type change, nullable change, default change) into one
  call. Review every parameter — a single `alter_column` can
  trigger several catalogue findings at once.
- `op.create_index(..., postgresql_concurrently=True)` requires
  the migration to run outside a transaction. Always confirm
  `transaction_per_migration = False` in `env.py` OR that the
  migration is split out. Recommending `postgresql_concurrently`
  without that setup will fail at runtime.
- Don't flag `op.execute("RAW SQL")` as inherently bad — it's
  often the *correct* choice for operations the builders can't
  emit. Review the SQL inside.
- A migration with `revision` and `down_revision` mismatched
  across the migration sequence in the PR (e.g. two migrations
  with the same `down_revision`) is ⚠️ CRITICAL — Alembic will
  refuse to apply both.
- If the project uses `alembic-utils` or similar for stateful
  database objects (views, functions, triggers), those objects'
  migrations look different — recognise them and apply the same
  catalogue.
