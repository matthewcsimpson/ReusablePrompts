# DB migration review — EF Core variant

Review database migrations generated by Entity Framework Core.

**This prompt extends [`core/db-migration-review.core.prompt.md`](./core/db-migration-review.core.prompt.md).**
Read the core first for the workflow shape (Step 0 convention
sourcing, Step 1 inventory, Step 2 unsafe-operation catalogue, Step
3 severity model, Step 4 report format, and the Constraints). This
file supplies the EF-Core-specific file paths, parsing strategy,
and ORM gotchas.

If pasting into a chat without filesystem access, paste the core
first, then this variant.

---

## Assumed stack

- **ORM**: Entity Framework Core 6+ (works for 5 as well; some
  surface-level details differ).
- **Migrations**: C# classes in `Migrations/` (or as configured
  via `--output-dir` on `dotnet ef migrations add`). Each class
  extends `Migration` and overrides `Up(MigrationBuilder)` /
  `Down(MigrationBuilder)`.
- **Schema source of truth**: `DbContext` subclass + entity types
  (POCOs with attribute or fluent API configuration). The model
  snapshot lives in `Migrations/<DbContext>ModelSnapshot.cs`.
- **Database provider**: from the `DbContext` `OnConfiguring` /
  `Program.cs` (`UseSqlServer`, `UseNpgsql`, `UseMySql`,
  `UseSqlite`).

EF Core's migrations are model-driven: changes to the
`DbContext` / entity classes drive `dotnet ef migrations add`,
which generates the migration class plus updates the model
snapshot. A migration without a corresponding model-snapshot
update is wrong.

## §1 — Inventory commands

```sh
# Migration classes in this PR
git diff --name-only origin/main...HEAD -- 'Migrations/*.cs' '**/Migrations/*.cs'

# Model snapshot — must update in lockstep
git diff --name-only origin/main...HEAD -- '**/Migrations/*ModelSnapshot.cs'

# Entity / DbContext changes
git diff origin/main...HEAD -- '**/*DbContext.cs' '**/Entities/*.cs' '**/Models/*.cs'

# Database provider detection
grep -rnE 'Use(SqlServer|Npgsql|Postgres|MySql|Sqlite|Oracle|Cosmos)' --include='*.cs' .

# Connection string hints
grep -rnE 'Server=|Host=|Data Source=' --include='*.json' --include='*.cs' . | head -n5
```

A migration with no corresponding model snapshot update is
⚠️ CRITICAL — EF Core uses the snapshot to compute future
migrations; the next `migrations add` will produce garbage.

## §2 — Operation catalogue (EF Core specifics)

The core's catalogue applies. EF Core's `MigrationBuilder` exposes
typed operations:

### Common `migrationBuilder.*` calls and their SQL

| Call | SQL emitted | Catalogue category |
|---|---|---|
| `AddColumn<T>(name, table, nullable: false)` | `ALTER TABLE ... ADD ... NOT NULL` — fails on populated table without default. |
| `AlterColumn<T>(name, table, nullable: false)` | `ALTER TABLE ... ALTER COLUMN ... NOT NULL` — table scan + lock. |
| `AlterColumn<T>(name, table, type: ...)` | `ALTER TABLE ... ALTER COLUMN ... TYPE ...` — may rewrite. |
| `DropColumn(name, table)` | `ALTER TABLE ... DROP COLUMN` — rolling-deploy breaker. |
| `RenameColumn(name, table, newName)` | `ALTER TABLE ... RENAME COLUMN` (or provider-specific equivalent) — rolling-deploy breaker. |
| `CreateIndex(name, table, columns)` | `CREATE INDEX` (no `CONCURRENTLY` from the typed API). |
| `AddForeignKey(...)` | `ALTER TABLE ... ADD CONSTRAINT FOREIGN KEY` — fails on bad data; FK index check needed. |
| `Sql("RAW SQL")` | Escape hatch; review by the core catalogue. |

### `Sql()` raw SQL escapes

For Postgres `CREATE INDEX CONCURRENTLY` (and any other operation
the typed API can't express), the pattern is:

```csharp
protected override void Up(MigrationBuilder migrationBuilder)
{
    migrationBuilder.Sql(
        "CREATE INDEX CONCURRENTLY IF NOT EXISTS \"IX_Users_Email\" " +
        "ON \"Users\" (\"Email\");",
        suppressTransaction: true);
}
```

The `suppressTransaction: true` argument is essential for
`CONCURRENTLY` (Postgres refuses inside a transaction). Without it,
the migration will fail at runtime.

If you see `migrationBuilder.Sql("CREATE INDEX CONCURRENTLY ...")`
without `suppressTransaction: true`, flag as ⚠️ HIGH — it will
error out partway through deployment.

### `AddColumn` with `defaultValue` vs `defaultValueSql`

`defaultValue: someConstant` embeds the literal in the migration —
applied once at creation. `defaultValueSql: "..."` emits a
server-side `DEFAULT` clause that persists in the schema.

Both shapes can trigger a table rewrite on some
provider / version combinations (notably MySQL < 8.0,
SQL Server pre-2016 for some types). Flag and check the provider.

### `Restrict` / `Cascade` / `SetNull` on FK actions

The default `OnDelete` behaviour varies by EF Core version and
configuration. A migration that adds a FK with `Cascade` against a
table with millions of dependent rows is a footgun — a single
parent delete becomes a long-running operation under lock. Flag
`Cascade` adds for review on hot tables.

### Discriminator / TPH / TPT / TPC changes

EF Core's table-per-hierarchy / table-per-type changes can emit
non-obvious migrations: `Discriminator` column adds, table moves,
data migration via `Sql("INSERT INTO ... SELECT ...")`. These are
correct in EF Core but warrant careful review for size of affected
data.

### `MigrationBuilder.UpdateData` and `InsertData`

EF Core's seed-data feature emits `UpdateData` and `InsertData`
calls inside the migration. These run as part of the migration
transaction — a large seed becomes a long-running operation.

For anything beyond a handful of reference rows, recommend moving
seeding out of migrations.

## §3 — Model snapshot integrity

For every migration in the PR:

- The model snapshot (`<DbContext>ModelSnapshot.cs`) must change in
  the same commit. Verify it.
- The snapshot must reflect the *cumulative* model state, not just
  this migration's changes. If the diff on the snapshot is smaller
  than expected (e.g. only one property changed but this migration
  modifies several entities), something is off.
- A snapshot diff that contains changes *not* present in the
  migration is ⚠️ HIGH — usually means an earlier migration was
  edited or skipped.

## §4 — `Down()` review

The core's "Rollback / downgrade review" applies. EF Core specifics:

- ⚠️ **Empty `Down()`** — body is empty or commented out.
  Migration is forward-only by negligence.
- ⚠️ **`Down()` references constraints by name where `Up()`
  generated them implicitly** — runtime failure on rollback.
- 💡 **`Down()` skips data restoration after an `Up()` that
  changed data shape** — schema reverses, data does not.

EF Core's tooling generates a default `Down()` from the model
delta, but heavily-edited migrations often have hand-written `Down`s
that drift. Confirm symmetry.

## Constraints (EF Core-specific addenda)

- A migration without a corresponding `<DbContext>ModelSnapshot.cs`
  update is ⚠️ CRITICAL. The snapshot is the basis for the next
  migration's diff; an outdated snapshot will produce a wrong
  migration on the next `dotnet ef migrations add`.
- Don't recommend "use the typed API" as a remediation for raw SQL
  — sometimes raw SQL is the only path (`CREATE INDEX
  CONCURRENTLY`, partial indexes, generated columns). The
  remediation is to review the SQL by the core catalogue.
- `migrationBuilder.Sql(..., suppressTransaction: true)` is the
  Postgres-specific escape for concurrent operations. For SQL
  Server, `WITH ONLINE = ON` requires Enterprise Edition; flag any
  use of it that the project's SKU doesn't support.
- A `Cascade` `OnDelete` action added against an existing FK
  relationship between hot tables is its own ⚠️ HIGH — the
  operational consequences extend far beyond the migration itself.
- The migrations directory should not contain manually-created
  files that aren't generated by `dotnet ef migrations add` —
  hand-authored migrations are legitimate but break the snapshot
  workflow if not also paired with snapshot edits.
