# DB migration review — Prisma variant

Review database migrations generated by Prisma Migrate.

**This prompt extends [`core/db-migration-review.core.prompt.md`](./core/db-migration-review.core.prompt.md).**
Read the core first for the workflow shape (Step 0 convention
sourcing, Step 1 inventory, Step 2 unsafe-operation catalogue, Step
3 severity model, Step 4 report format, and the Constraints). This
file supplies the Prisma-specific file paths, parsing strategy, and
ORM gotchas.

If pasting into a chat without filesystem access, paste the core
first, then this variant.

---

## Assumed stack

- **ORM**: Prisma (`@prisma/client`, `prisma` CLI).
- **Schema source of truth**: `prisma/schema.prisma`.
- **Migrations**: `prisma/migrations/<timestamp>_<name>/migration.sql`
  — raw SQL generated by `prisma migrate dev`.
- **Database**: detect from the `datasource` block in
  `schema.prisma`. Prisma supports PostgreSQL, MySQL, SQLite,
  SQL Server, MongoDB, CockroachDB; this review covers the
  relational engines.

Prisma does not have a `down` migration concept. Rollback is not
something Prisma performs; the project handles rollback by writing
a forward migration that reverses the change. Note this in the
report — the "Rollback / downgrade review" section reads
differently for Prisma than for ORMs with `down()` methods.

## §1 — Inventory commands

```sh
# New migrations in this PR (compared to merge base)
git diff --name-only origin/main...HEAD -- 'prisma/migrations/**/migration.sql'

# Or all migrations on this branch not on main
git log origin/main..HEAD --name-only --format='' \
  | grep '^prisma/migrations/.*/migration.sql$' | sort -u

# Datasource (engine detection)
grep -A3 '^datasource' prisma/schema.prisma

# Schema delta for context (what the migration is supposed to express)
git diff origin/main...HEAD -- prisma/schema.prisma
```

The schema delta is essential context. The migration SQL is what
*will run*; the schema delta is what the author *intended*. Many
findings come from divergence between intent and SQL — e.g. the
schema renames a field via `@map`, but the migration `DROP`s and
`ADD`s the column, losing data.

## §2 — Operation catalogue (Prisma specifics)

The core's catalogue applies directly — the migration is raw SQL.
A few Prisma-specific patterns to watch for:

### `@map` / `@@map` rename → drop+add

Prisma's introspection sometimes turns a model / field rename into
**`DROP COLUMN` + `ADD COLUMN`** rather than `RENAME COLUMN`. This
loses every row's data for that column.

The fix is to hand-edit the generated migration to use `ALTER TABLE
... RENAME COLUMN` instead. If you see paired `DROP COLUMN x` and
`ADD COLUMN y` with the same type, and the `schema.prisma` delta
shows a `@map("x")` annotation added to a renamed field, this is
⚠️ CRITICAL.

### Implicit relation tables

Prisma's implicit many-to-many (no explicit join model) creates a
table named `_ModelAToModelB`. A schema change that reorders
participants in the relation can prompt a `DROP TABLE _AtoB` +
`CREATE TABLE _BtoA` — silently losing the relationship data.

### `prisma db push` artefacts

If `prisma db push` was used during development, the resulting SQL
may include destructive operations that `prisma migrate dev` would
have prompted about. Check for un-named migrations or migrations
with `_db_push_` in the name.

### `prisma migrate resolve --applied`

A migration marked `--applied` is recorded in `_prisma_migrations`
without being run. If this PR includes a `migration_lock.toml`
change or modifies `_prisma_migrations` directly, flag as ⚠️ HIGH
— it bypasses the migration safety net.

### Native database features Prisma doesn't track

Prisma's schema doesn't model database-level constraints like
trigger functions, generated columns (until recently), partial
indexes, or custom types. If the migration adds these via
`Unsupported("...")` or raw SQL, the schema and the database
diverge — future migrations may attempt to reconcile them in
unexpected ways.

### `previewFeatures`

If the migration was generated under a preview feature flag, the
SQL may include experimental shapes that change behaviour between
Prisma versions. Note any preview features active in
`schema.prisma`.

## §3 — Concurrent index workaround

Prisma migrations run inside a single transaction by default.
Postgres refuses `CREATE INDEX CONCURRENTLY` inside a transaction.

The Prisma pattern for concurrent indexes:

1. Generate the migration with `--create-only`:
   `prisma migrate dev --create-only --name add_x_idx`.
2. Edit the migration SQL: remove the `BEGIN` / `COMMIT` (Prisma
   wraps the file in a transaction unless told otherwise), or
   split the index creation into its own migration.
3. Document the split.

If you see a `CREATE INDEX` (no `CONCURRENTLY`) on what looks like
a large or hot table, recommend the `--create-only` path with
explicit instructions, not just "use CONCURRENTLY."

## §4 — Rollback section (Prisma-specific phrasing)

Prisma has no `down`. In the report:

```
## Rollback / forward-only reversal

Prisma migrations are forward-only. To "roll back" this migration,
the project will need to:

- For `ADD COLUMN`: write a new migration that `DROP`s the column
  (loses any data written in the meantime).
- For `DROP COLUMN`: not recoverable from the migration system —
  restore from backup.
- For type changes: write a new migration that converts back, with
  the same caveats about precision loss.

Confirm the team is comfortable with forward-only semantics for
this change. If the migration is high-risk and forward-only,
recommend running it against a clone of production first.
```

## Constraints (Prisma-specific addenda)

- Don't recommend hand-editing `migration.sql` without warning
  that the edit must happen before any `prisma migrate dev` /
  `prisma migrate deploy` runs — once applied, the recorded
  checksum will mismatch and CI will refuse to continue.
- A `--create-only` generated migration that has *not* been
  applied locally yet is the right place to make changes; one
  that has been applied requires a fresh migration on top.
- If the PR modifies `schema.prisma` but does not add a migration
  file, flag as ⚠️ HIGH — Prisma will not pick up schema changes
  in production without an explicit migration.
- Prisma's `shadow database` requirement during local migration
  development can hide real issues that surface only in
  production engines (e.g. RDS Postgres has different
  `max_locks_per_transaction` defaults). Mention this caveat if
  the migration uses many DDL operations in one file.
